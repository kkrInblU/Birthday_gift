## 一、已解决问题

### 1. MySQL 连接认证失败

问题现象：

- 程序启动时出现数据库认证失败

处理结果：

- 已修正数据库连接配置
- 当前建库、建表、入库、查询均可正常执行

### 2. 定时任务只执行一轮

问题现象：

- 已配置调度间隔和轮数，但任务只执行一次

处理结果：

- 已厘清调度开关与调度参数关系
- 多轮定时抓取验证已完成

### 3. 网络请求阶段性失败

问题现象：

- 曾出现 SSL、连接中断、WinError 10013 等错误

处理结果：

- 已通过分层排查完成环境与链路验证
- 抓取链路可正常执行

### 4. 附件下载链路不稳定

问题现象：

- OA 附件下载依赖会话和页面上下文，无法稳定直接对外暴露下载地址

处理结果：

- 已收敛方案：
  - 不保存附件下载地址
  - 不在小程序内直接下载附件
  - 前端仅展示附件名称、格式、大小
  - 点击附件时复制原文链接，引导用户前往 OA 原页面下载

### 5. 正文富文本在小程序中排版异常

问题现象：

- OA 原始 HTML 在小程序 `rich-text` 中显示效果差，图片与文字排版混乱

处理结果：

- 已重构小程序专用正文 HTML 生成逻辑
- 正文图片与段落显示已恢复可读性

### 6. 前端与后端联调异常

问题现象：

- 曾出现 `url not in domain list`、接口未及时更新、详情页卡住等问题

处理结果：

- 本地 API 联调链路已稳定
- 小程序首页、详情页、我的页当前可正常读取接口数据

### 7. 用户刷新周期设置链路

处理结果：

- 已完成 `users` 表字段扩展：
  - `notification_refresh_interval_minutes`
  - `last_notification_check_at`
- 已完成接口：
  - `GET /api/users/settings`
  - `POST /api/users/settings`
- 已完成小程序“我的”页真实接入
- 已确认规则：
  - 用户每次修改刷新周期，固定重置 `last_notification_check_at = 当前时间`

### 8. 小程序提醒主流程自动发送

这是之前遗留的问题，目前已解决。

之前状态：

- 主投递流程只会创建 `miniapp` 投递记录
- 不会自动调用微信订阅消息发送接口
- 因此数据库中会出现较多 `miniapp = pending` 的记录

当前状态：

- `oa_delivery_main.py` 已接入小程序自动发送链路
- 投递任务会自动处理待发送的 `miniapp` 记录
- 会真实调用微信订阅消息发送接口
- 会把发送结果回写到 `notification_delivery_log`

状态含义：

- `pending`：已入投递队列，但尚未执行发送
- `success`：已成功发送
- `failed`：已执行发送，但发送失败

已验证结果：

- 订阅部门新通知可自动触发小程序发送
- `miniapp` 投递记录可从 `pending` 更新为 `success`
- `provider_message_id` 已验证可正常落库

### 9. 历史订阅数据与当前规则不一致

问题现象：

- 历史上部分 `subscriptions.enable_email = 0`
- 与当前“订阅部门后邮件默认开启”的业务规则冲突

处理结果：

- 当前数据已核对修正
- 数据库初始化流程已补充历史数据修正规则

### **需求补充清单**

**1. 用户角色与权限的详细定义**

1、本科生、研究生：筛选本科生、研究生信息，本科生身份可以只看本科生相关信息，研究生可以只看研究生相关信息。
2、学生/教师：可以只看教职工相关信息（同上）
3、管理员：进行系统爬虫配置、管理、巡检（待后续补充）

**如何判定相关？目前我的想法是正文内容包含本科生、研究生、教职工，则认为与对应人群相关，但是这个方法比较粗糙，有没有更好的办法？**

**3. 订阅管理功能的完整用户需求**

* * **取消订阅**：用户取消对某个部门的订阅需要二次确认？
  * **批量订阅/取消**：支持一次性订阅/取消多个部门？
  * **订阅渠道单独编辑**：用户不可以针对某个部门，单独设置只接收邮件提醒，或只接收小程序提醒？
  * **订阅列表展示**：在订阅管理页面，按照目前设计展示用户已订阅的部门

**4. 搜索功能的详细需求**

* **现状**：文档提到了“关键词快速查找历史通知”、“模糊匹配标题和正文”、“按发布时间倒序展示结果”、“搜索历史”。这些是初步需求。
* **补充需求分析**：

  * **搜索范围**：支持用户选择搜索的范围是搜索全部通知，还是仅限用户已订阅的通知
  * **高级搜索**：是否支持按发布单位、时间范围、通知分类等进行筛选？不做，可以放到TODO里面，后续有时间补充。
  * **搜索结果排序**：按照发布时间由新到旧排序，支持相关性排序。（**这里的相关性算法是否与区分本科生、研究生、教职工新闻的是同一套？**）
  * **搜索性能**：对搜索响应时间是否有具体要求？（暂时不做要求，可记录一下）
  * **搜索历史管理**：支持清除搜索历史

**5. 收藏功能的详细需求**

* **现状**：文档提到了“收藏列表”、“取消收藏”、“持久化存储”、“分页加载”。
* **补充需求分析**：

  * **收藏数据的同步策略**：目前前端是本地存储，如果用户在不同设备登录，收藏列表需要同步，所以后端要实现持久化接口
  * **收藏后的通知状态变化**：收藏的通知按照目前前端样式进行展示不做更改

**6. 消息提醒模块的详细需求**

* **补充需求分析**：

**7. 用户个人信息管理需求**

* **现状**：文档提到了“查看、修改个人信息”、“修改手机号、邮箱、密码”。
* **补充需求分析**：

  * **修改手机号/邮箱的验证流程**：新手机号/邮箱需要验证码验证，验证码的有效期是5分钟和重发需要等待180s.
  * **密码修改的安全性要求**：需要旧密码验证，新密码强度要求与旧密码一样

  **1. 管理员登录**

  * **思考方向**：如何确保只有授权人员才能访问后台？登录方式的安全性、便捷性如何平衡？
    * 管理的登录入口在登陆页面的下方（管理员登录），点击后输入管理员账号密码可进入管理员界面
  * **需求点**：
    * **登录方式**：

      * 需要独立的管理员账号体系（用户名+密码）
    * **会话管理**：

      * 登录仅单次有效，会话没有缓存，退出小程序后需要重新登陆。
    * **安全性**：

      * 登录失败次数上限是3次
      * 登录日志记录登录时间和地点

  **2. 任务管理（爬虫调度）**

  * **思考方向**：管理员需要对爬虫任务的运行状态有清晰的了解，并能在必要时进行干预。
  * **需求点**：
    * **任务列表**：
      * 展示所有爬虫任务（例如：OA 通知抓取、邮件投递、小程序消息投递）。
      * 显示任务状态（运行中、已完成、失败、暂停）。
      * 显示任务类型、上次运行时间、下次计划运行时间。
    * **手动触发**：
      * 管理员是否可以手动触发某个任务立即执行（例如：手动触发一次 OA 通知抓取）。
      * 手动触发时是否可以指定参数（例如：抓取特定栏目、抓取最近 N 条）。
    * **暂停/恢复调度**：
      * 管理员是否可以暂停或恢复全局的爬虫调度。
      * 是否可以暂停或恢复某个特定任务的调度。
    * **任务配置**：
      * 管理员是否可以在后台修改爬虫的配置参数（例如：`SCHEDULER_INTERVAL_MINUTES`、`MAX_RECORDS`、`REQUEST_DELAY_MIN/MAX`）。
      * 修改后是否需要立即生效或重启服务？
    * **历史记录**：
      * 查看 `crawl_job_log` 的详细记录，包括每次任务的开始/结束时间、耗时、抓取/入库数量、状态、错误信息。

  **3. 日志查看**

  * **思考方向**：管理员需要快速定位问题、了解系统运行细节。日志是重要的诊断工具。
  * **需求点**：
    * **日志类型**：
      * 系统运行日志（爬虫、API、投递服务的 INFO/WARNING/ERROR 日志）。
      * 操作日志（管理员在后台执行了什么操作）。
      * 异常日志（所有捕获到的异常信息）。
    * **筛选功能**：
      * 按时间范围筛选。
      * 按日志级别筛选（INFO, WARNING, ERROR）。
      * 按模块筛选（爬虫、API、投递）。
      * 按关键词搜索。
    * **展示方式**：
      * 分页展示，支持排序。
      * 日志详情查看（点击查看完整堆栈信息）。
    * **日志导出**：
      * 是否支持将日志导出为文件（例如 CSV, TXT）。

  **4. 系统监控**

  * **思考方向**：管理员需要对系统的整体健康状况、性能指标和关键业务数据有宏观的了解。
  * **需求点**：
    * **核心指标**：
      * **爬虫**：最近一次抓取时间、抓取成功率、平均抓取耗时、每日/每周抓取总量。
      * **数据库**：通知总数、附件总数、用户总数、订阅总数。
      * **API**：API 请求总量、平均响应时间、错误率。
      * **投递**：邮件/小程序消息发送总量、成功率、失败率。
    * **健康状态**：
      * 数据库连接状态。
      * API 服务运行状态。
      * 调度器运行状态。
    * **告警机制**：
      * 当关键指标异常（例如：爬虫连续失败、API 错误率过高）时，如何通知管理员（邮件、短信、微信）。
  * **可视化**：
      * 是否需要简单的图表展示趋势（例如：每日新增通知数量趋势图）。

## 四、2026-04-10 人群筛选相关待优化问题

### 1. 教务处规则可能带来误判

当前状态：

- 本科生筛选规则中，`教务处` 被作为部门命中规则参与加分。
- 该规则可以提升本科生通知的召回率，但也可能把部分面向全校、教师或其他对象的通知误判为本科生相关。

影响：

- 会影响本科生筛选结果的准确性。
- 若前端后续直接按本科生标签展示专区列表，误判会直接影响用户体验。

### 2. 当前是规则匹配，不是语义理解

当前状态：

- 当前本科生、研究生、教职工三类筛选，均基于标题、分类、正文、发布单位中的显式关键词与部门规则进行匹配。
- 系统目前不能理解通知语义、适用对象、上下文范围，只能做基于规则的近似判断。

影响：

- 对于“未明确写出关键词，但实际面向某类人群”的通知，可能漏判。
- 对于“关键词出现但并非主要面向该类人群”的通知，可能误判。

### 3. 规则表虽然已入库，但后台管理界面还没做

当前状态：

- 规则表 `audience_keyword_rules`、`audience_department_rules` 已经建立并投入使用。
- 但当前仍缺少后台页面，无法在界面上直接维护：
  - 关键词
  - 权重
  - 部门映射
  - 启停状态
  - 规则版本

影响：

- 当前规则调整仍依赖开发改代码或直接改数据库，维护成本较高。
- 后续答辩或交付阶段，不利于展示规则可运营、可维护能力。

### 4. 需要后续补重算机制、人工校验机制

当前状态：

- 当前已具备代码级“对现有通知重算人群标签”的能力。
- 但还没有正式后台入口、任务化管理、重算日志，也没有人工核验流程。

影响：

- 当规则版本升级后，历史通知标签修正成本较高。
- 缺少人工校验机制，不利于沉淀误判样本、漏判样本和规则调优记录。

建议：

- 后续补充全量重算、指定范围重算、重算日志。
- 后续补充人工抽样核验和问题样本登记机制。

### 3. 当前爬虫任务管理能力还不够细

当前状态：

- 已支持查看配置、保存配置、手动触发单次抓取、查看最近任务日志。
- 但还没有暂停调度、恢复调度、取消任务、指定栏目抓取、指定数量抓取等能力。

影响：

- 当前更像“最小可运行管理入口”，还不是完整运维面板。

### 4. 当前只覆盖了抓取任务日志，未覆盖完整后台巡检能力

当前状态：

- 当前日志查询基于 `crawl_job_log`，主要面向抓取任务结果。
- 尚未纳入投递任务、系统异常、管理员操作日志等后台巡检数据。

影响：

- 管理员目前只能看到抓取侧结果，问题定位能力仍不完整。

### 5. 当前运行时配置是“后续任务生效”，不是对所有常驻进程强制热更新

当前状态：

- 当前数据库配置已能驱动新启动的抓取任务读取最新参数。
- 但若后续存在长期常驻的独立调度进程，还需要明确热更新或重载策略。

影响：

- 配置即时生效这一点，目前对“接口触发的新任务”成立；对未来常驻服务形态，仍需补工程化约束和说明。

影响：

- 当前更适合开发联调和功能验证。
- 距离长期稳定运行仍有工程化缺口。
